Sesión 7.2. Integración de servicios sobre móviles

Ingeniería de Servicios

2023-2024

1 Introducción

En esta sesión se integrarán varios servicios sobre una aplicación móvil. Por un lado se accederá a la información almacenada en la aplicación web “amigos” desarrollada en sesiones previas. Además, se usarán los servicios de OpenStreetMap para obtener mapas del mundo y la biblioteca osmdroid para pintarlos en una aplicación Android, de modo que la posición de los amigos se pueda visualizar de forma gráfica sobre un mapa. Por último, se usarán servicios de notificación para enviar información desde el servidor a los clientes y así evitar el polling.

2 Pintar amigos en el mapa

Google proporciona un conjunto de servicios web que pueden ser utilizados en tus propias aplicaciones. Uno de estos servicios es el servicio de mapas. Al ser Android un operativo desarrollado también por Google, la integración entre ambos servicios es muy sencilla. De hecho, Android Studio incluye un tipo de aplicación “Mapa” que automatiza por completo la creación de una especie de “hola mundo” de los mapas, que después se podría extender para añadirle otra funcionalidad.

Este podría ser un excelente punto de partida para nuestra práctica (y de hecho así se hacía otros años), pero para poder hacer uso de la API de mapas era necesario obtener antes una API_KEY que identificara al desarrollador concreto (y al entorno de desarrollo que se usara para hacer la aplicación), sin la cual el mapa aparecerá en blanco. Esta API_KEY se podía obtener de forma gratuita a través de la consola para desarrolladores de Google (bastaba tener una cuenta Google), pero desde fechas muy recientes Google te obliga a dar detalles de facturación asociados a tu cuenta si quieres obtener dicha API_KEY. Aunque el acceso sigue siendo relativamente gratuito (Google te “regala” 200$ para uso de sus APIs de pago), lo cierto es que se pide al desarrollador que introduzca una tarjeta de crédito válida para verificar que no es un robot.

Debido a lo anterior, hemos decidido cambiar de proveedor de mapas para esta sesión, aunque ello pueda suponer un desarrollo algo menos simple que con las APIs de Google. No te preocupes, en lo que sigue se detalla cómo instalar y usar osmdroid, para lo cual no se requiere ninguna API_KEY y por tanto ningún registro.

2.1 Configuración de la librería de desarrollo

Para poder desarrollar en Android aplicaciones que usen los servicios de OpenStreetMap es necesario indicar la dependencia de la libreria osmdroid, como detallaremos después.

Además, al final de esta práctica, se utilizarán también servicios de Google para recibir notificaciones en el teléfono, lo que requiere además que el operativo del dispositivo tenga instalada la librería Google Play Services.

Para comprobar si está instalada, abre el Android SDK Manager y comprueba en la pestaña SDK Tools que la librería Google Play Services está instalada.

Además, para poder probar las aplicaciones que usen esa librería, es necesario disponer bien de un teléfono real, o bien de un emulador que incluya una versión del operativo con los binarios de Play Store. El emulador que instalamos en la práctica anterior (Nexus 5X con API 24) lo tenía, como puedes comprobar.

Entra en el Device Manager y observa que en target aparece la cadena “Google APIs”. También puedes comprobar en el Android SDK Manager, pestaña SDK Platforms, marcando la opción Show Package Details que en el apartado Android 7.0 (Nougat) está instalado el componente Google Play Intel x86 Atom System Image o Google APIs Intel x86 Atom_64 System Image. Este es el que correrá en el emulador.

2.2 Creación de un proyecto con mapas de OpenStreetMap

2.2.1 Preparación

  • Crea un nuevo proyecto en Android. Elige la plantilla “Empty Views Activity” (la misma que ya usamos para el conversor de divisas)

  • El nombre de la aplicación será Amigos y, por tanto, el nombre del paquete sea es.uniovi.amigos. Elige como minimum SDK la API 19 y selecciona el lenguaje Java, al igual que hicimos en la sesión anterior, y deja el resto de opciones por defecto.

  • Genera la aplicación pulsando Finish, y espera a que finalice

Para que la aplicación pueda utilizar la biblioteca omsdroid es necesario especificarla entre las dependencias del proyecto. Para ello abre el fichero build.gradle.kts (Module: app) y añade al final, entre las dependencies la siguiente:

    implementation("org.osmdroid:osmdroid-android:6.1.13")

Además será necesario que la aplicación “declare” ciertos permisos para poder acceder al GPS y a los archivos del dispositivo. Para declararlos edita el AndroidManifest.xml y añade al final, detrás del tag </application> lo siguiente (tras ello, recuerda pulsar File -> Sync Project with Gradle Files para descargar los nuevos componentes de la aplicación):

   <uses-sdk android:targetSdkVersion="NEWEST AVAILABLE" android:minSdkVersion="8" />
   <uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION"/>
   <uses-permission android:name="android.permission.ACCESS_FINE_LOCATION"/>
   <uses-permission android:name="android.permission.ACCESS_WIFI_STATE" />
   <uses-permission android:name="android.permission.ACCESS_NETWORK_STATE" />
   <uses-permission android:name="android.permission.INTERNET" />
   <uses-permission android:name="android.permission.WRITE_EXTERNAL_STORAGE" />

2.2.2 Código mínimo

Si intentaras compilar y ejecutar ahora mismo la aplicación, no deberías encontrar errores (puedes probar), pero tampoco verás ningún mapa, pues el código de la aplicación y su layout son todavía los del “Hola mundo” que Android Studio genera como demostración para una Empty Views Activity.

Necesitamos modificar este esqueleto por defecto para que en su lugar muestre un mapa.

  • Abre el fichero res > layout > activity_main.xml

  • En lugar de la vista Design, accede a la vista Code, donde verás el XML que define el aspecto de la aplicación. Cámbialo todo por este otro:

    <?xml version="1.0" encoding="utf-8"?>
<LinearLayout xmlns:android="http://schemas.android.com/apk/res/android"
    android:orientation="vertical"
    android:layout_width="fill_parent"
    android:layout_height="fill_parent">
    <org.osmdroid.views.MapView android:id="@+id/map"
        android:layout_width="fill_parent"
        android:layout_height="fill_parent" />
</LinearLayout>

    Como ves el layout se compone ahora de un solo elemento de tipo org.osmdroid.views.MapView que será donde se mostrará el mapa.

También es necesario cambiar el código de la aplicación. Edita MainActivity.java y reemplaza todo el código que aparece por el siguiente:

package es.uniovi.amigos;
import android.Manifest;
import android.content.Context;
import android.content.pm.PackageManager;
import android.os.Bundle;
import android.preference.PreferenceManager;

import androidx.appcompat.app.AppCompatActivity;
import androidx.core.app.ActivityCompat;
import androidx.core.content.ContextCompat;

import org.osmdroid.config.Configuration;
import org.osmdroid.tileprovider.tilesource.TileSourceFactory;
import org.osmdroid.views.MapView;

import java.util.ArrayList;

public class MainActivity extends AppCompatActivity{
    private final int REQUEST_PERMISSIONS_REQUEST_CODE = 1;
    private MapView map = null;  // Este atributo guarda una referencia al objeto MapView
                                 // a través del cual podremos manipular el mapa que se muestre

    @Override
    public void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        // Leer la configuración de la aplicación
        Context ctx = getApplicationContext();
        Configuration.getInstance().load(ctx, PreferenceManager.getDefaultSharedPreferences(ctx));

        // Crear el mapa desde el layout y asignarle la fuente de la que descargará las imágenes del mapa
        setContentView(R.layout.activity_main);
        map = (MapView) findViewById(R.id.map);
        map.setTileSource(TileSourceFactory.MAPNIK);

        // Solicitar al usuario los permisos "peligrosos". El usuario debe autorizarlos
        // Cuando los autorice, Android llamará a la función onRequestPermissionsResult
        // que implementamos más adelante
        requestPermissionsIfNecessary(new String[]{
                // WRITE_EXTERNAL_STORAGE este permiso es necesario para guardar las imagenes del mapa
                Manifest.permission.WRITE_EXTERNAL_STORAGE
        });
    }

    @Override
    public void onResume() {
        super.onResume();
        map.onResume();
    }

    @Override
    public void onPause() {
        super.onPause();
        map.onPause();
    }

    @Override
    public void onRequestPermissionsResult(int requestCode, String[] permissions, int[] grantResults) {
        // Esta función será invocada cuando el usuario conceda los permisos
        // De momento hay que dejarla como está
        super.onRequestPermissionsResult(requestCode, permissions, grantResults);
        ArrayList<String> permissionsToRequest = new ArrayList<>();
        for (int i = 0; i < grantResults.length; i++) {
            permissionsToRequest.add(permissions[i]);
        }
        if (permissionsToRequest.size() > 0) {
            ActivityCompat.requestPermissions(
                    this,
                    permissionsToRequest.toArray(new String[0]),
                    REQUEST_PERMISSIONS_REQUEST_CODE);
        }
    }

    private void requestPermissionsIfNecessary(String[] permissions) {
        // Itera por la lista de permisos y los va solicitando de uno en uno
        // a menos que estén ya concedidos (de ejecuciones previas)
        ArrayList<String> permissionsToRequest = new ArrayList<>();
        for (String permission : permissions) {
            if (ContextCompat.checkSelfPermission(this, permission)
                    != PackageManager.PERMISSION_GRANTED) {
                // Permission is not granted
                permissionsToRequest.add(permission);
            }
        }
        if (permissionsToRequest.size() > 0) {
            ActivityCompat.requestPermissions(
                    this,
                    permissionsToRequest.toArray(new String[0]),
                    REQUEST_PERMISSIONS_REQUEST_CODE);
        }
    }
}

3 Integración de servicios web de la aplicación amigos

Una vez que tenemos la posibilidad de mostrar un mapa, vamos a integrar sobre la aplicación el servicio desarrollado en la sesión sobre aplicaciones web. Esto implica hacer una consulta http a nuestro servidor flask (en concreto a la ruta que retornaba la información en JSON), parsear correctamente la respuesta y usar la API de mapas para pintar “chinchetas”. La primera parte, es decir la consulta vía HTTP y parseo del JSON, es análoga a la que ya hicimos en la aplicación “Converter”, y requiere del uso de la biblioteca Volley (aunque ahora será necesaria la versión 1.2.0 de dicha librería).

3.1 Haciendo la aplicación “amigos” accesible al mundo

La aplicación la tenemos hecha de la sesión 2.2. Entra en la máquina virtual y en la carpeta de esa sesión para lanzarla. Recuerda que habíamos “contenedorizado” el despliegue de modo que era necesario lanzar tres contenedores, uno con la base de datos MariaDB, otro con el servidor web nginx y otro más con la aplicación Flask. Probablemente tengas scripts para simplificar el lanzamiento de esos contenedores, como se explicó en la práctica correspondiente.

Lanza la aplicación y verifica que está funcionando (puedes conectar tu navegador desde windows a la URL http://ip.de.tu.ubuntu/html/amigos para ver la versión HTML con la tabla, o a http://ip.de.tu.ubuntu/api/amigos) para ver la versión JSON.

Si todo ha ido bien, la aplicación estará accesible para cualquier ordenador en la misma subred que la máquina Ubuntu, pero no para el resto del mundo. Si vas a querer probar la aplicación móvil que estamos desarrollando desde un teléfono móvil real, el teléfono no podrá acceder al servidor porque está en otra subred (típicamente el teléfono tendrá conexión a internet a través de una red de datos 3G/4G, o de Wifi).

Podemos usar el siguiente truco para que la aplicación sea visible al mundo entero.

  • Instala en Ubuntu ngrok (puedes hacerlo con sudo snap install ngrok)
  • Ejecuta en la terminal Ubuntu el comando ngrok http 80
  • En la pantalla aparecerá una URL del estilo http://1395db1c124d.ngrok.io (es diferente cada vez que ejecutes ngrok). Esta URL es visible desde cualquier lugar del mundo, y cuando un navegador se conecte a ella, ngrok redirigirá la petición a tu máquina Ubuntu, aunque la tengas en modo NAT y aunque esté ejecutándose detrás de un router NAT.
  • Deja esa terminal funcionando (pues si la cierras, la redirección dejará de funcionar también)

Comprueba que funciona, conectando un navegador a esa URL. Pídele a tu compañero/a que intente conectar a esa URL también desde su casa para ver que funciona desde cualquier lugar. Puedes probar también desde tu móvil, usando la conexión de datos en vez de Wifi y debería funcionar también.

Esta redirección no es muy estable ni admite mucho tráfico, y se considera que debes usarla sólo para pruebas como ésta y no para tener un servidor local en producción. Si pulsas Ctrl-C, el túnel se rompe y el servidor deja de estar accesible al mundo. Lanzando el mismo comando puedes rehacer el túnel (pero la URL pública será otra).

3.2 Consumo de los datos desde la aplicación móvil

Ya hemos visto en la sesión anterior cómo usar Volley para pedir datos JSON a un servicio Web. Lo usaremos ahora para obtener la lista de amigos.

Crea una clase Amigo para contener la información de un amigo, con atributos para guardar su nombre, latitud y longitud (no necesitas guardar su id de dispositivo).

Añade a la clase principal de la aplicación un atributo de tipo List<Amigo>, para guardar la lista de amigos que vamos a recibir. Crea un método getAmigosList() que se conecte a la URL apropiada (conviene que esta URL sea otro atributo de la clase, para que sea fácil cambiarla) para obtener la lista de amigos JSON usando Volley.

Ya que la respuesta del servidor en este caso es una lista el tipo de petición para Volley habrá de ser JsonArrayRequest, y el tipo de la respuesta recibida será JSONArray, del cual se pueden extraer los objetos individuales (los amigos y sus coordenadas) mediante el método .getJSONObject(int). El parámetro int es el índice dentro del array del elemento que queremos obtener. Podemos iterar por todos los elementos en un bucle. El array tiene un método .length() para obtener cuántos elementos son.

De cada uno de los elementos de ese array puedes extraer el nombre con .getString("name") y las coordenadas con .getDouble("lati") y .getDouble("longi"). Con esta información instanciarás un objeto Amigo y lo añadirás a la lista de amigos.

3.3 Pintando chinchetas

El pintado de las “chinchetas” debe hacerse en el hilo principal. Puedes implementar un método llamado por ejemplo paintAmigosList() que lo haga, pero ese método debe ser llamado desde el hilo principal.

Hay un problema de asincronía aquí. Para poder pintar la lista de amigos necesitamos por un lado tener la propia lista (que solicitamos vía HTTP pero no sabemos cuándo llegará), y por otro lado que el mapa haya terminado de pintarse, lo cual es otra operación asíncrona pues osmdroid va pidiendo a los servidores de OpenStreetMaps las imágenes de cada “baldosa” con las que va componiendo el mapa.

Ambas solicitudes son asíncronas y no sabemos cuál ocurrirá antes. Cuando recibamos la lista se ejecutará el método onResponse() que has implementado como parte del listener para Volley. Si intentamos pinchar las chinchetas desde dentro de onResponse() puede que el mapa aún no esté listo. Este tipo de problemas aparece muy a menudo en el ámbito de la programación asíncrona, hasta el punto de que se han desarrollado nuevos paradigmas y sintaxis para tratar con ello (como RxJava).

En esta sesión optaremos por una solución simple, y daremos por supuesto que, para cuando se reciba la lista de amigos, el mapa estará ya terminado de pintar, por lo que se podrán dibujar las chinchetas encima de él. En el caso de omsdroid, además, parece que esta hipótesis no plantea problemas porque, aún si el mapa no se había terminado de dibujar, es posible añadirle chinchetas, que aparecerán más tarde cuando el mapa se dibuje.

Para crear una marca en el mapa (una chincheta) con omsdroid implementaremos la siguiente función:

    private void addMarker(double latitud, double longitud, String name) {
        GeoPoint coords = new GeoPoint(latitud, longitud);
        Marker startMarker = new Marker(map);
        startMarker.setPosition(coords);
        startMarker.setAnchor(Marker.ANCHOR_CENTER, Marker.ANCHOR_CENTER);
        startMarker.setTitle(name);
        map.getOverlays().add(startMarker);
    }

Este código hace uso de clases implementadas por omsdroid, como GeoPoint para contener la latitud y longitud de un punto del mapa, y Marker para representar la “chincheta” con una cierta forma y “nombre” asociado (será el nombre del “amigo” que se halla en esa localización).

La función paintAmigosList() deberá iterar sobre la lista de amigos (que recibirá como parámetro) y para cada uno llamar a la función addMarker() que acabamos de ver.

3.4 Actualización de las posiciones de las chinchetas

Además de pintar la posición de los amigos al arrancar la aplicación, deberíamos repintarlos cada cierto tiempo, pues los contenidos de la base de datos pueden cambiar. Para ello, de momento, crearemos un temporizador como el siguiente (donde UPDATE_PERIOD es el tiempo deseado entre actualizaciones, en milisegundos, que puedes fijar a 5000 por ejemplo para que se ejecute cada 5 segundos):

Timer timer = new Timer();
TimerTask updateAmigos = new UpdateAmigosPosition();
timer.scheduleAtFixedRate(updateAmigos, 0, UPDATE_PERIOD);

UpdateAmigosPosition es el nombre de una clase (declarada dentro de la clase principal de la aplicación, para que tenga acceso a sus métodos), que ha de contener un método llamado run, desde el que invocaremos la actualización de la lista y el repintado de las chinchetas. Es decir:

class UpdateAmigosPosition extends TimerTask {
    public void run() {
        getAmigosList();
    }
}

Añade la actualización periódica al programa. ¿Dónde crees que deberías poner el código que crea el temporizador y lo prepara para que lance la actualización de la lista cada cierto tiempo?

Nota. La función paintAmigosList() deberá comenzar por borrar las chinchetas antes pintadas, pues de no hacerlo así cada nueva invocación a esta función pintaría chinchetas adicionales encima de las que ya había. Para borrar las chinchetas previas usa map.getOverlays().clear(); Esta misma situación ocurre con la lista de amigos en getAmigosList(), donde deberás vaciar tu lista con el método clear() antes de añadir todos los amigos de la respuesta, o de lo contrario se añadirían nuevos Amigos en lugar de actualizar los existentes.

Nota 2. Debido a un posible bug de omsdroid, aunque el código de pintar chinchetas se ejecute de nuevo, la vista del mapa no se actualiza hasta que el usuario no interactúe de nuevo con ella, por ejemplo tocando la pantalla del móvil. Para evitar este problema añade la siguiente línea una vez has pintado a todos los amigos: map.getController().scrollBy(0,0); Esto simulará una acción del usuario (desplazar el mapa, si bien lo desplazamos cero unidades), y eso forzará su repintado, con lo que se verá inmediatamente la actualización de las chinchetas tras su repintado.

Fíjate que estamos realizando la actualización mediante polling, lo cual nunca es eficiente, ya que no se puede saber si hay datos más actuales antes de realizar la conexión. Más adelante se solucionará ese problema con la integración de los servicios de notificaciones.

Comprueba que el mapa se repinta correctamente cada cierto tiempo. Una forma sencilla de verificarlo es usar un navegador web para acceder a la tabla de amigos en http://ip.de.ubuntu/html/amigos y editar las coordenadas de uno de ellos, o añadir o borrar amigos de la tabla. El mapa debería actualizar las chinchetas adecuadamente.

Opcional. Puedes mejorar la apariencia de la información cambiando las chinchetas que omsdroid pinta por defecto por otro icono. Para ello puedes usar startMarker.setIcon(getResources().getDrawable(R.drawable.icono)); en la función addMarker(), pero el icono debe existir en la lista de recursos de la aplicación. Esa lista la puedes ver en el árbol del proyecto, res > drawable. Por defecto sólo aparecen ahí dos archivos, correspondientes al icono de la aplicación. Puedes añadir más mediante File > New > Image Asset (o Vector Asset). Al añadir uno de la categoría “Launcher Icons (Legacy only)” tendrás acceso a muchos iconos predefinidos en el menú “Clip Art”.

4 Actualización de la posición en función del GPS

De momento la única forma en que las chinchetas se pueden mover es si accedes al servidor mediante su interfaz html. Vamos a hacer algo más realista. Se supone que cada amigo lleva un teléfono en el que está corriendo esta aplicación. Por tanto, la aplicación tomará de sensor GPS del teléfono cuáles son sus coordenadas y las enviará a la base de datos, cada vez que se detecte que el teléfono se ha movido.

En una aplicación real cada usuario debería estar identificado por un login y contraseña, debería darse de alta de algún modo, y debería identificarse al instalar la aplicación. Para simplificar vamos a omitir el registro y la necesidad de contraseña. Al arrancar la aplicación se pedirá un nombre al usuario y se usará ese nombre para acceder a la base de datos y actualizar la posición de ese usuario. Se entiende que el nombre proporcionado debe estar ya en la base de datos (lo habrás introducido antes a través de su interfaz HTML, por ejemplo).

4.1 Permisos para acceder a la posición

Para que la aplicación pueda acceder al sensor GPS es necesario obtener los permisos apropiados.

Añade al manifiesto los permisos necesarios para acceder a la posición del GPS en caso de que no lo estuvieran ya.

<!-- Se require para obtener la posición -->
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />

4.2 Obtener el nombre del usuario

Nos hace falta saber cuál es el nombre de la persona cuya posición queremos actualizar. Se lo podemos preguntar al usuario de la aplicación en OnCreate llamando a la siguiente función que crea un diálogo (observa que en este caso el layout lo creamos directamente desde java, en lugar de leerlo de un XML):

public void askUserName() {
      AlertDialog.Builder alert = new AlertDialog.Builder(this);

       alert.setTitle("Settings");
       alert.setMessage("User name:");

       // Crear un EditText para obtener el nombre
       final EditText input = new EditText(this);
       alert.setView(input);

       alert.setPositiveButton("Ok", new DialogInterface.OnClickListener() {

4.3 Obtención de la posición

La obtención de la posición de un dispositivo móvil es una tarea compleja, ya que hay que adaptarse al ciclo de vida de la aplicación.

En un caso sencillo se puede obtener la posición de la siguiente forma:

  • Añade, en el código java, método onCreate(), el permiso Manifest.permission.ACCESS_FINE_LOCATION a la lista de permisos que se le pasan a la función requestPermissionsIfNecessary(), para que cuando se ejecute la aplicación se solicite al usuario este nuevo “permiso delicado”, además del de acceder al almacenamiento.
  • Añade, al final de la función onCreate() un llamada a SetupLocation(), función que implementaremos seguidamente

La función SetupLocation() llamada desde el código anterior se ocupa de registrar un listener al Location Manager. Ese listener será invocado cada vez que el GPS detecte un cambio de coordenadas, o cada cierto tiempo, según como haya sido creado. Un ejemplo del código de esta función sería el siguiente:

void SetupLocation() {
    if (ActivityCompat.checkSelfPermission(this,
                              android.Manifest.permission.ACCESS_FINE_LOCATION)
            != PackageManager.PERMISSION_GRANTED
        && ActivityCompat.checkSelfPermission(this,
                              android.Manifest.permission.ACCESS_COARSE_LOCATION)
            != PackageManager.PERMISSION_GRANTED)
     {
        // Verificar por si acaso si tenemos el permiso, y si no 
        // no hacemos nada
        return;
    }


    // Se debe adquirir una referencia al Location Manager del sistema
    LocationManager locationManager =
          (LocationManager) this.getSystemService(Context.LOCATION_SERVICE);

    // Se obtiene el mejor provider de posición
    Criteria criteria = new Criteria();
    String  provider = locationManager.getBestProvider(criteria, false);

    // Se crea un listener de la clase que se va a definir luego
    MyLocationListener locationListener = new MyLocationListener();

    // Se registra el listener con el Location Manager para recibir actualizaciones
    // En este caso pedimos que nos notifique la nueva localización
    // si el teléfono se ha movido más de 10 metros
    locationManager.requestLocationUpdates(provider, 0, 10, locationListener);

    // Comprobar si se puede obtener la posición ahora mismo
    Location location = locationManager.getLastKnownLocation(provider);
    if (location != null) {
        // La posición actual es location
    } else {
        // Actualmente no se puede obtener la posición
    }
}

Finalmente hay que implementar el listener, lo que nos obliga a implementar una serie de métodos de esta interfaz, aún si no los necesitamos (los dejamos vacíos en ese caso):

        double longi = location.getLongitude();

        // para actualizar en el backend la posición de este teléfono
        // ...

    // El resto de métodos que debemos implementar los podemos dejar vacíos
    @Override
    public void onStatusChanged(String provider, int status, Bundle extras) {}
    // Se llama cuando se activa el provider
    @Override
    public void onProviderEnabled(String provider) {}
    // Se llama cuando se desactiva el provider

5 Servicios de notificaciones

Los servicios de notificaciones permiten enviar información desde los servidores a los clientes. Estos servicios denominados Push evitan que los clientes tengan que solicitar información al servidor sin saber si esa información está disponible (el mecanismo Pull). El mecanismo Push es más eficiente, ya que reduce el intercambio de información, pero es más difícil de implementar. En concreto se trabajará con el servicio Firebase Cloud Messaging (FCM). Este servicio permite enviar información desde un servidor a clientes Android y también a aplicaciones sobre Chrome. Otros fabricantes, como Apple o Microsoft, tienen servicios equivalentes.

El servicio Firebase Cloud Messaging (FCM) permite que servidores propios, por ejemplo el servidor Flask de la aplicación amigos, envíe información a aplicaciones que se ejecutan sobre Android, iOS o clientes Web. Además, esa información se transmite a los clientes Android utilizando conexiones existentes de otros servicios de Google, con lo que el tráfico de información resulta muy eficiente.

Los elementos que participan en el intercambio de mensajes son los siguientes:

  • Servidor de la aplicación (Trusted Environment en la figura) es el servidor que contiene la información que los clientes necesitan. En nuestro caso será la aplicación amigos (también se pueden enviar mensajes a los dispositivos a través de una consola web que Firebase proporciona, útil para pruebas).

  • Cliente: es la aplicación Android que usa la información del servidor (también puede ser una aplicación iOS o Web)

  • Servidores FCM: son los servidores proporcionados por Google que se encargarán de hacer llegar el mensaje desde el servidor de la aplicación hasta la aplicación cliente.

5.1.1 Obtener token del dispositivo

Cuando el servidor flask quiera enviar un mensaje a un dispositivo concreto, necesitará conocer su token (un identificador único). Es para esto que habíamos incorporado a la base de datos “amigos” un campo “device”, que de momento está aún en blanco para todos los usuarios (o debería estar).

Debemos incluir código para que, cuando nuestra app arranque, obtenga un token de dispositivo.

  • Edita el manifest y añade dentro de <application> un nuevo servicio:

    <service
    android:name=".MyFirebaseMessagingService"
    android:exported="false">
    <intent-filter>
        <action android:name="com.google.firebase.INSTANCE_ID_EVENT"/>
        <action android:name="com.google.firebase.MESSAGING_EVENT" />
    </intent-filter>
</service>

    Esto declara un nuevo componente de la aplicación, de tipo servicio. Pero aún tenemos que implementarlo (mientras no lo hagamos este fichero nos mostrará en rojo el nombre del servicio).

  • Crea una nueva clase llamada MyFirebaseMessagingService (el mismo nombre que hemos puesto en el atributo android:name de este servicio en el manifest). Esta clase debe heredar de FirebaseMessagingService e implementar el método onNewToken(), así por ejemplo:

    public class MyFirebaseMessagingService extends FirebaseMessagingService {
    @Override
    public void onNewToken(String token) {
        System.out.println("FCM Refreshed token: " + token);
    }
}

Cuando arranca la aplicación, arrancará también este servicio que acabamos de crear, el cual creará un token único para el dispositivo en que se está ejecutando. En este caso nos limitamos a imprimir por el logcat el token en cuestión.

5.1.2 Registro del token dispositivo ante nuestro servidor

El token mencionado en el ejercicio anterior es el valor que deberemos guardar en la base de datos del backend, en el campo device asociado con el usuario cuyo nombre solicitamos al arrancar la app. Para ello habría que enviárselo a la aplicación Flask, mediante un PUT al usuario apropiado. El problema es que seguramente no sabemos aún quién es el usuario, y que probablemente tengamos el token mucho antes de que el usuario haya escrito su nombre, pues el token se genera casi inmediatamente durante el arranque (y sólo la primera vez que se ejecuta la aplicación y no vuelve a generarse de nuevo a menos que la aplicación sea reinstalada).

Para obtener el valor del token en cualquier otro momento (por ejemplo, desde la Actividad principal del mapa), será necesario un código como el siguiente:

public void getFirebaseToken() {
    FirebaseMessaging.getInstance().getToken()
            .addOnCompleteListener(new OnCompleteListener<String>() {
                @Override
                public void onComplete(@NonNull Task<String> task) {
                    String token = task.getResult();
                    System.out.println("FCM getFirebaseId() ->" + token);
                }
            });
}

Se trata de código asíncrono. El método getToken() retorna una promesa que se resolverá cuando llegue la respuesta. Mediante .addOnCompleteListener() se añade a la promesa una función para que se ejecute cuando la respuesta llegue. La función es onComplete() y el valor que recibe como parámetro es la promesa resuelta (task). Invocando su método getResult() obtendremos el valor resuelto de la promesa, es decir, el Token que Firebase nos haya asignado.

En este ejemplo nos limitamos a imprimirlo en el logcat, pero puedes guardarlo en una variable miembro de la clase actividad, para usarlo cuando el usuario haya introducido su nombre. Una vez se tenga el token se usará como parte de un PUT /api/amigos/{id} para enviar un JSON que contenga ese token en el campo "device".

public class MyFirebaseMessagingService extends FirebaseMessagingService {
    @Override
    public void onMessageReceived(RemoteMessage remoteMessage) {
        System.out.println("FCM: From " + remoteMessage.getFrom());

        // El mensaje en sí son datos que pueda enviar el servidor
        // en forma de cadena que contiene una especie de JSON
        // No lo usaremos, por lo que vendrá vacío
        if (remoteMessage.getData().size() > 0) {
            System.out.println("FCM: Message = " + remoteMessage.getData());
        }

        // La notificación es el texto que aparecería en la barra
        // de notificaciones, el que se especifica en la consola
        // web de Firebase
        if (remoteMessage.getNotification() != null) {
            System.out.println("FCM: Notification Body ="
                    + remoteMessage.getNotification().getBody());
        }
    }
}

5.1.4 Actualizar posiciones de amigos sólo cuando es necesario

Elimina o comenta el código que creaba un temporizador para ejecutar getAmigosList() periódicamente. Haz que se ejecute sólo una vez, cuando el programa arranca. Comprueba que la aplicación arranca y pinta las chinchetas en el mapa una vez (la terminal en que ejecutas ngrok te muestra las peticiones HTTP que está recibiendo el backend, comprueba que sólo recibe una solicitando la lista de amigos).

La idea, tras eliminar el polling, es que el cliente invoque getAmigosList() sólo cuando le llegue un mensaje a través de FCM. El cuerpo del mensaje o el texto de la notificación que recibamos es irrelevante en este caso. Simplemente por recibir un mensaje asumiremos que algo ha cambiado en la base de datos, por lo que solicitaremos la nueva lista de amigos y la repintaremos.

El problema es que la recepción del mensaje ocurre en el servicio implementado por la clase MyFirebaseInstanceService (en un hilo distinto al del GUI), pero la invocación de getAmigosList() debe hacerse desde la clase MainActivity() (en el hilo del GUI).

Necesitamos por tanto transmitir el evento de que se ha recibido un mensaje de un componente a otro de nuestra aplicación. Para esta situación Android tiene los Broadcast Receivers.

En el componente MyFirebaseMessagingService, dentro de la función onMessageReceived() añade el código siguiente:

    Intent updateFromServer = new Intent("updateFromServer");
    LocalBroadcastManager.getInstance(this).sendBroadcast(updateFromServer);

Esto crea un intent (que es una estructura de datos frecuentemente usada en Android para transmitir información entre componentes, entre otras cosas). La cadena "updateFromServer" sirve para identificar a este intent cuando sea recibido en otra parte de la aplicación, y así diferenciarlo de otros intents y poder tomar diferentes acciones. Seguidamente envía ese intent a través del LocalBroadcastManager. Esto hará que los restantes componentes de la aplicación (en particular, la actividad principal que pinta el mapa) lo reciban, siempre que hayan registrado una clase para recibir estos mensajes.

Para que otro componente reciba el mensaje debes instanciar un objeto de la clase BroadcastReceiver, que implemente el método onReceive(). Después debe registrar ese objeto a la vez que un filtro de intents, para que el objeto no reciba cualquier intent, sino sólo los que encajen con ese filtro.

Por ejemplo, el siguiente código registra un receptor de mensajes para los intent cuya acción sea "updateFromServer", que es la que se usó en la creación del intent en el código antes visto.

mHandlerForBroadcast = new BroadcastReceiver() {
    @Override
    public void onReceive(Context context, Intent intent) {
        System.out.println("Broadcast: received = " + intent.getAction());
    }
};
LocalBroadcastManager.getInstance(this).registerReceiver(
        mHandlerForBroadcast,
        new IntentFilter("updateFromServer"));

5.1.5 Detalle final

Aunque lo anterior funcione, no es del todo correcto. La función que hemos añadido a la actividad principal registra un componente que queda a la escucha de broadcast, pero no lo desregistra nunca. Si la actividad muere (por ejemplo, el teléfono gira), el componente seguirá esperando mensajes. Al re-crear la actividad (por ejemplo, tras un giro del teléfono) lo registraríamos de nuevo. Si giramos el teléfono varias veces podríamos acabar con varios componentes escuchando por mensajes broadcast, y todos ellos se activarían ante el intent "updateFromServer" (lanzando por tanto varias peticiones al backend).

Para evitarlo, hay que des-registrar el componente que escucha cuando la aplicación pasa al modo Paused, y registrarlo de nuevo si pasa por el modo Resumed. Los métodos siguientes serían para eso:

    @Override
    protected void onResume() {
        super.onResume();
        map.onResume(); //needed for compass, my location overlays, v6.0.0 and up
        LocalBroadcastManager.getInstance(this).registerReceiver(
                mHandlerForBroadcast,
                new IntentFilter("updateFromServer"));
    }

    @Override
    protected void onPause() {
        LocalBroadcastManager.getInstance(this)
                             .unregisterReceiver(mHandlerForBroadcast);
        super.onPause();
        map.onPause();  //needed for compass, my location overlays, v6.0.0 and up
    }

5.2 Integración de FCM en el backend

La parte del cliente está terminada. El último paso consiste en que, en lugar de generar las notificaciones desde la consola web de Firebase, lo hagamos desde la aplicación Flask cuando la base de datos sea modificada.

La idea es que cuando nuestro backend reciba una actualización de coordenadas GPS (porque el teléfono de alguno de los usuarios se ha movido más de 10 metros y por tanto ese teléfono ha hecho un PUT a la base de datos), entonces Flask se ponga en contacto con los servidores FCM y les pida que manden la notificación a la aplicación. Ya hemos visto que cuando ésta recibe la notificación solicita la lista actualizada, por lo que el objetivo final estaría completado.

5.2.1 Enviando notificaciones desde python

Para enviar una notificación usando FCM desde nuestro servidor Ubuntu podemos usar Python (o cualquier lenguaje de programación). Basta saber que los servidores de Firebase proporcionan una API REST, y cuáles son las cabeceras y datos que hay que pasarles con las peticiones que les hagamos. Esta API está descrita en la documentación de Firebase.

Ni siquiera necesitamos conocer esos detalles si usamos una biblioteca que los implemente por nosotros, como es el caso de la pyfcm en el caso de Python.

Instala pyfcm en el entorno virtual en que teníamos instalado Flask (es decir, activa ese entorno y ejecuta pip install pyfcm dentro de él)

Un sencillo ejemplo de código python que envía una notificación a nuestro teléfono usando FCM sería el siguiente:

from pyfcm import FCMNotification

push_service = FCMNotification(
                    api_key="AIza...JQtE")
registration_id = "eBzYFzwyID0:APA91...vhW0"
message_title = "Hola desde Python!"
message_body = "Este mensaje ha sido enviado desde python"

result = push_service.notify_single_device(
                        registration_id=registration_id,
                        message_title=message_title,
                        message_body=message_body)
print(result)

Debes averiguar cuáles son los valores correctos a rellenar para la api_key y el registration_id, en la forma siguiente:

  • La api_key la obtienes en la consola Firebase. Dentro del proyecto “Amigos”, en el menú de la izquierda justo debajo del logo de Firebase hay un icono con una tuerca. Púlsalo y elige “Configuración del proyecto” y después ve a la pestaña “Cloud Messaging”. Allí verás una sección para la (V1) Habilitada y para la (heredada) deshabilitada.

    Para el uso con pyfcm necesitamos la versión (heredada) por lo que deberás pulsar en los tres puntos de la derecha, abrir el enlace que aparece y pulsar sobre “Habilitar” en la web que aparece. Una vez hecho, refresca la web de Firebase y podrás ver tu clave:

    La larga cadena que comienza por AAA es la clave que debes copiar en api_key (No es buena política meter estas claves en el código, pues el código suele terminar en un repositorio git y por tanto la clave ser visible para quienes tengan acceso al repo. Lo correcto sería que el script Python tomara la api_key de una variable de entorno, pero este script de prueba no lo vamos a poner bajo control de versiones, de modo que puedes dejarlo así).

  • El registration_id es el id del dispositivo, el cual pudiste ver en el logcat de la ejecución de la aplicación, o también en la base de datos (interfaz web de amigos), pues es el valor del campo “device” para el usuario.

Si completas el código anterior y lo ejecutas, el teléfono con el id suministrado debería recibir un mensaje. Si la aplicación “Amigos” no se está ejecutando en ese teléfono, el mensaje se verá en la barra de notificaciones. Si se está ejecutando, en cambio, no se verá porque el mensaje será consumido por la aplicación, pero deberías poder verlo en el logcat de Android Studio, pues lo estábamos imprimiendo allí cuando llegaba.

Prueba el código anterior y verifica que el mensaje llega al teléfono.

Nota. Ahora sí, el mensaje deberá llegar al teléfono de forma casi instantánea, ya que no es la consola FireBase quien lo envía y quien a veces introduce retrasos. La aplicación Python lo envía al servidor FireBase y éste lo reenvía inmediatamente al teléfono con el token suministrado.

Se pueden enviar mensajes a varios dispositivos en una sola llamada, si tienes una lista con los id de los dispositivos en cuestión. Así:

registration_ids = [ 
        "eBzYFzwyID0:APA91...vhW0",
        "zxjkasloHS1:APA21...1jkv",
        "wJbasTUajH1:IQU92...uYsh"
        ]
result = push_service.notify_multiple_devices(
                        registration_ids=registration_ids,
                        message_title=message_title,
                        message_body=message_body)
