1 Introducción

El objetivo de esta práctica es el desarrollo de una aplicación Web mínima y el estudio del protocolo HTTP que permite intercambiar información entre el cliente y el servidor Web. Para el desarrollo de la aplicación se utilizará el framework Flask instalado en la sesión anterior y la arquitectura de tres contenedores (nginx, flask y mariadb) ya vista. Para el estudio del protocolo las herramientas del Firefox.

Vamos a desarrollar una pequeña aplicación Web que sirva para almacenar información sobre dónde se encuentran un conjunto de amigos. Básicamente contendrá una lista de nombres de amigos juntos con sus coordenadas en longitud y latitud. Para ello se deberá crear en una base de datos una tabla para contener la información y desarrollar una aplicación flask para servir esa información a través de la Web.

Para el desarrollo de la aplicación en esta práctica vamos a seguir el patrón modelo-vista-controlador. Este patrón permite diferenciar distintas partes de la aplicación como son la forma en la que se almacena la información (modelo de datos) de la forma en la que se muestra la información (vista, mediante templates). Por último, aparece un tercer componente que se encarga de enlazar vista y modelo e interactuar con el cliente, denominado controlador, que implementa una serie de funciones que se invocarán según la ruta solicitada por el cliente para manipular el modelo y generar una vista como respuesta. El controlador puede generar diferentes vistas, por ejemplo una en HTML para ser mostrada a un usuario humano en un navegador y otra en JSON o XML para ser consumida por otro tipo de aplicación, por ejemplo una aplicación móvil.

3.1 Estructura básica de la aplicación Flask

La aplicación “amigos” es más compleja que la “hola flask” que habíamos hecho en la sesión anterior, por lo que adoptaremos desde el principio buenas prácticas y la escribiremos separada en varios ficheros con una estructura que la haga más mantenible.

Es costumbre meter el código fuente de los diferentes ficheros que compondrán la aplicación en una carpeta llamada app. El contenido de esta carpeta irá bajo control de versiones. Todos los valores que sirvan para configurar diferentes aspectos de la aplicación (por ejemplo, si se ejecuta en modo debug o no, el puerto en que escucha si no queremos que sea el usado por defecto, etc.) se guardan en un archivo config.py fuera de esa carpeta. Es posible definir diferentes conjuntos de configuracione, por ejemplo uno para usar durante el desarrollo, otro durante el testing y otro para producción. Todo eso iría dentro de config.py, el cual va también bajo control de versiones.

Por otro lado, ya que la aplicación tendrá que conectarse en algún momento a una base de datos, la clave para conectar con ella es necesaria. Incluir esa clave como parte del código fuente no es buena idea ¡especialmente si el repositorio git donde guardamos todo es público, como sería el caso de un proyecto open-source!. Una buena práctica consiste en tener toda la información sensible (secretos) en un archivo aparte, fuera de la carpeta app y que no esté bajo control de versiones. También es habitual que esa información esté en variables de entorno que puedan ser consultadas desde python (y esto es especialmente útil en escenarios con contenedores ya que hemos visto que con la opción -e podríamos pasarles esas variables de entorno).

De momento crearemos una carpeta llamada instance donde poner archivos que son específicos del entorno donde será lanzada la aplicación, y dentro de esa carpeta config.py definirá la contraseña de acceso a la base de datos, por ejemplo.

Esta será de momento la estructura de carpetas y algunos de los archivos que iremos creando después:

├── amigos/
       ├── app/
       │   ├── __init__.py
       │   ├── templates/
       │   ├── models.py
       │   └── views.py
       ├── config.py
       └── instance/
           └── config.py

3.1.1 Configuración

El archivo amigos/config contendrá lo siguiente:

# Configuraciones para distintas formas de lanzar la aplicación

class Config:
    "Clase base de la que derivarán las demás"
    # Pueden ponerse valores comunes a todos los entornos de ejecución
    SQLALCHEMY_TRACK_MODIFICATIONS = False

class DevelopmentConfig(Config):
    "Configuración de desarrollo"
    DEBUG = True
    SQLALCHEMY_ECHO = True


class ProductionConfig(Config):
    "Configuración de producción"
    DEBUG = False


# Diccionario que asocia un nombre con una configuración
app_config = {
        'development': DevelopmentConfig,
        'production': ProductionConfig
        }

Se sale de los objetivos de la práctica explicar en detalle estas opciones. Son una especie de “constantes” que serán usadas por Flask o por SQLAlchemy. En concreto, SQLALCHEMY_ECHO sirve para que, cada vez que la aplicación intente acceder a la base de datos, se muestren por consola los comandos SQL que se están enviando a la misma.

3.1.2 Conexión con la base de datos

La conexión la gestionará SQLAlchemy, un paquete python que se integra con flask a través del paquete flask-sqlalchemy que instalamos en el entorno virtual.

Para ello necesita conocer la “url” para acceder a la base de datos. Esta url tendrá la forma general:

driver://usuario:contraseña@nodo/database

En nuestro caso el driver será mysql+pymysql, que indica que conectaremos con una base de datos MySQL (o MariaDB), vía el módulo pymsql. El usuario sería amigosuser, la contraseña amigospass y la base de datos amigosdb, pero… ¿cuál es el nodo? La base de datos se está ejecutando en un contenedor docker, por lo que el nodo sería la IP de ese contenedor.

Como de momento el desarrollo lo estamos haciendo en el anfitrión, necesitamos conocer la IP exacta del contenedor. Cuando todo esto se ejecute en otro contenedor podremos usar el DNS de docker y entonces el nodo sería simplemente "basedatos", pues ese es el nombre del contenedor en que está el servidor mariadb.

Averigua la IP del contenedor basedatos con el comando docker inspect basedatos

Una vez sepamos la IP, podremos crear un archivo python que defina la URL a la base de datos. Ya que esa URL incluye un secreto (la contraseña de acceso a amigosdb), no podemos ponerlo en el fichero global de configuración, ya que éste está bajo control de versiones y la clave sería visible en el repositorio.

Edita el fichero instance/config.py y escribe lo siguiente:

# Definición de secretos a usar por la app
SQLALCHEMY_DATABASE_URI = "mysql+pymysql://amigosuser:amigospass@<IP>/amigosdb"

Donde <IP> es la que has averiguado.

3.1.3 Mini-prueba

Vamos a hacer de nuevo un “hola mundo”, pero más complejo que el de la sesión anterior, porque éste debe usar la infrastructura de carpetas y archivos de configuración que estamos creando, y ha de conectarse con la base de datos (aún si todavía no la utiliza para nada).

La idea es tener una factoría de aplicaciones. Una factoría de aplicaciones en este caso es una función que crea y retorna un objeto de la clase Flask. Esta función se ocupa de configurar la aplicación creada, haciendo uso de las configuraciones que tenemos en amigos/config.py, y de conectarla con la base de datos, haciendo uso de la URL que definimos en amigos/instance/config.py. Una vez ha creado y configurado la app, retorna el objeto así creado.

Todo esto lo implementamos en el archivo amigos/app/__init__.py que contendría por tanto lo siguiente:

from flask import Flask
from flask_sqlalchemy import SQLAlchemy

# Importamos el diccionario de configuraciones (production, development)
from config import app_config

# Crear el objeto db que servirá para conectar con la base de datos
db = SQLAlchemy()

# La factoría recibe como parámetro que configuración queremos usar
def create_app(config_name):
    # La app simplemente es una instancia de la clase Flask
    # La "receta" obliga a pasarle el nombre del módulo en que fue creada
    # que python guarda en la variable __name__. El parámetro
    # instance_relative_config es para decirle que cuando leamos
    # ficheros de configuración, lo haga de la carpeta ./instance
    app = Flask(__name__, instance_relative_config=True)

    # Una vez creada, la app debe almacenar una configuración
    # La sacamos del diccionario de configuraciones
    app.config.from_object(app_config[config_name])

    # Extendemos esa configuración con la URL de la base de datos
    # que leemos de otro fichero de configuración en ./instance
    app.config.from_pyfile('config.py')

    # Pasamos la aplicación ya configurada a db.init(), quien usará
    # la URL para conectar con la base de datos sql
    db.init_app(app)

    # Configuremos una ruta de prueba para la app
    @app.route("/")
    def prueba():
        return "¡Hola Flask con MariaDB!"

    return app

El nombre de fichero __init__.py es especial para Python. Tan pronto como una carpeta contiene un fichero con ese nombre, la carpeta puede ser usada en un import. Es decir, desde otro script podemos hacer import app y eso causará la ejecución de app/__init__.py, con lo que después podríamos acceder a app.create_app() o app.db. También podemos hacer from app import X siendo X cualquier función o variable definidos de forma global en app/__init__.py. Por ejemplo, from app import create_app o from app import db.

Como ves, además de crear la app y configurarla, la factoría también la usa como decorador para registrar una función (prueba()) para una ruta "/", y así podremos hacer desde un cliente una petición http GET / y ver si obtenemos la respuesta. No es muy limpio definir las rutas dentro de la factoría y no es como suele hacerse. Más adelante lo corregiremos pero de momento nos vale para hacer pruebas.

Para poder lanzar la aplicación, ya sea desde el servidor de desarrollo flask run o desde gunicorn u otro servidor wsgi, necesitamos otro módulo que invoque a la función create_app() y que guarde el resultado en una variable global (al módulo) llamada app, pues eso es lo que necesita un servidor wsgi.

Así que escribimos el siguiente código en amigos/lanzar.py:

# Importar la función create_app desde la carpeta app
from app import create_app

# Usarlo para instanciar la aplicacion
app = create_app("development")

  1. Escribe los dos ficheros anteriores.

  2. Probemos a lanzar la aplicación desde línea de comandos en la forma siguiente:

    $ FLASK_APP=lanzar.py flask run -h 0.0.0.0

  3. Conecta un navegador con la IP de tu máquina Ubuntu, puerto 5000 y verifica que recibes el mensaje de Flask.

Si lo anterior ha ido bien ¿qué ha ocurrido exactamente? Es interesante pararse a comprender bien todos los detalles:

  • Al usar flask run, éste ha mirado la variable de entorno FLASK_APP para saber qué script debería cargar. Ha cargado por tanto lanzar.py, el cual comienza cargando a su vez app/__init__.py y así se define la función create_app(). Después se ha llamado a create_app() que ha creado la aplicación y ha registrado la ruta "/", retornando la app creada y guardándose el resultado en una variable también llamada app (no confundir con la carpeta app). Flask espera que exista esa variable pues es con ella con la que establece el protocolo wsgi.
  • Observa no obstante que, como parte de la configuración de la app, se han leido los archivos de configuración ./config.py y instance/config.py cuyos valores han quedado registrados en la variable app.
  • También como parte de la creación de la app se ha llamado a db.init(), quien intentará establecer una conexión con la base de datos cuya URL espera en la variable SQLALCHEMY_DATABASE_URI de la app.
  • Puesto que no hemos visto errores, eso implica que la conexión con la base de datos ha tenido éxito. Recordemos que la base de datos se está ejecutando en el contenedor docker llamado basedatos.
  • Después Flask queda a la espera de peticiones de clientes, en el puerto 5000 de la máquina Ubuntu.
  • Cuando el navegador se conecta a ese puerto, la conexión tiene éxito (es Flask quien responde). El navegador envía por la conexión el comando GET / y las cabeceras pertinentes.
  • Flask recibe esa petición, analiza la ruta a la que va dirigida y comprueba entre las rutas registradas de la variable app cuál es la función encargada de responder para esa ruta, encontrando que se trata de la función prueba().
  • Flask ejecuta esa función (en este caso sin pasarle parámetros, pero podría pasárselos como veremos después) y espera la respuesta. La función en este caso retorna una cadena.
  • Flask empaqueta el valor retornado en una respuesta HTTP válida, añadiendo las cabeceras apropiadas y se lo envía al cliente a través de la conexión que tenía abierta. Después típicamente cierra esa conexión.
  • El cliente (navegador) recibe una respuesta HTTP válida y la muestra.

Ya tenemos todo listo para comenzar la implementación de la aplicación “Amigos”. Ya que usaremos un modelo MVC, separaremos el desarrollo en esas tres partes: Modelo, Vista y Controlador.

3.2 Modelo

Tras la exitosa prueba inicial sabemos que nuestra app puede conectar con la base de datos. Sin embargo no sabe nada de cuál es la estructura de la información allí almacenada, por lo que no sabría cómo crear nuevas tablas, elementos dentro de las tablas o qué consultas SQL hacer.

El siguiente paso por tanto es definir el modelo de datos.

Nuestro modelo de datos será muy sencillo. La base de datos contendrá una única tabla llamada amigos que tendrá las siguientes columnas:

  • Un id numérico que actúe como clave primaria, autoincrementado
  • Un campo nombre con una cadena que será el nombre del amigo.
  • Dos campos lat y long que almacenarán la latitud y longitud del amigo (lo almacenaremos también como cadenas, para simplificar, aunque realmente son números de coma flotante)

Podríamos usar el cliente mariadb para conectar con la base de datos y crear manualmente esta tabla mediante comandos SQL, pero no lo haremos así porque el módulo sqlalchemy nos permite usar sólo python para definir los datos y nos evita tener que conocer los detalles de sintaxis SQL de la base de datos que usemos en particular. Observa que al conectar con la base de datos, la URL que usamos comenzaba por mysql:// lo que le dice a SQLAlchemy qué la base de datos es MySQL o una compatible como MariaDB. Si hubiera comenzado por postgres://, sabría que se trata de una PostgreSQL. En cualquier caso SQLAlchemy adaptará los comandos SQL que envíe a la base de datos al dialecto particular de la base de datos usada, liberando de estos detalles al programador.

Edita el fichero amigos/app/models.py y escribe en él lo siguiente:

from app import db

class Amigo(db.Model):
    """
    Definición de la tabla 'amigos' de la base de datos
    """

    __tablename__ = "amigos"

    # Lo siguiente define las columnas de la base de datos y sus tipos
    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(32), unique=True)
    longi = db.Column(db.String(32))
    lati = db.Column(db.String(32))

    # Podemos escribir la función siguiente para implementar cómo debe
    # mostrarse un objeto de esta clase si lo imprimes desde python
    def __repr__(self):
        return "<Amigo[{}]: {}>".format(self.id, self.name)

Usando SQLAlchemy el modelo de datos se implementa mediante clases. Cada clase corresponderá a una tabla en la base de datos. El código anterior define la clase Amigos que será la plantilla usada para crear objetos que se guardarán en la tabla amigos. Cada objeto tendrá como columnas los atributos definidos en el código anterior (id, name, longiy lati).

El código anterior define también un método __repr__(). No es imprescindible, pero es útil para depuración pues, si en algún momento hacemos un print() de un objeto de la clase Amigo, en lugar de mostrársenos la dirección en que se guarda el objeto, se mostrará algo más amigable, como es el ID del amigo y su nombre, por ejemplo <Amigo[1]: Manolo>.

3.2.1 Creación de la tabla en la base de datos

El modelo anterior permite recuperar información de la base de datos si ésta tuviera ya la tabla amigos con la estructura deseada. No es así, pero gracias a flask-migrate (otro de los paquetes que hemos instalado al principio con pip install) el propio modelo puede servir para crear las tablas pertinentes en la base de datos o para realizar migraciones.

3.2.2 Migraciones

Aunque se sale del temario de la asignatura y está más bien relacionado con bases de datos, explicaremos someramente que entendemos por migración. Imagina que llevas ya unos días trabajando en la aplicación, la base de datos tiene ya varios amigos con sus nombres y coordenadas, y entonces se te ocurre que querrías modificar la información de esos amigos, por ejemplo añadiéndoles otro campo como telefono. La base de datos no tiene ese campo. Necesitamos un nuevo esquema. Lo que tendrías que hacer es destruir la tabla amigos para volver a crearla con las nuevas columnas. Esto naturalmente te haría perder la información que ya había. Una migración sería el proceso de guardar una copia de lo existente, re-crear la base de datos con el nuevo esquema y volver a rellenar la copia con lo que había antes (salvo los campos que hayas añadido, claro, que se pondrían a NULL). De igual modo podrías querer eliminar campos.

Las migraciones son un proceso doloroso, por lo que intentan evitarse pensando muy bien el diseño de la base de datos antes de ponerse a implementar nada. Por desgracia a veces son inevitables, y aquí es donde flask-migrate puede ser de gran utilidad porque es capaz de detectar automáticamente qué ha cambiado en el esquema de la base de datos y de generar una serie de scripts python que automaticen la migración. Es más, va guardando un histórico de las versiones por las que pasa la base de datos y permite hacer tanto upgrades (pasar a la siguiente versión) como downgrades (volver a la anterior).

3.2.3 Primera migración

En realidad, si lo piensas, nos enfrentamos a una primera migración, pues pasamos de una versión de la base de datos en la que no tiene aún tablas, a otra en la que tendrá la tabla amigos. Podemos usar flask-migrate para que automatice esta migración y así cree las tablas necesarias, tal como las hemos definido en app/models.py.

Para ello:

  1. Modifica amigos/app/__init__.py para que soporte migraciones, como se muestra en el código siguiente:

    # parte de app/__init__.py
from flask_migrate import Migrate
# ...
def create_app():
    # ... todo igual que antes, pero al final...
    migrate = Migrate(app, db)
    from app import models
    return app

  2. Ejecuta (desde la carpeta amigos con el entorno virtual activado):

    $ FLASK_APP=lanzar.py flask db init

  3. Comprueba que ha aparecido una carpeta nueva llamada migrations, y dentro de ella versions. No necesitamos entrar en ella, es donde flask-migrations guarda sus scripts para hacer los upgrade o downgrade cuando se lo pidamos. Puedes ver que de momento está vacía.

  4. Ejecuta el siguiente comando:

    $ FLASK_APP=lanzar.py flask db migrate

  5. Comprueba que han aparecido ficheros en migrations/versions. Serán los ejecutados ante un upgrade o downgrade.

  6. Ejecuta el siguiente comando:

    $ FLASK_APP=lanzar.py flask db upgrade

  7. Repite el Experimento 1 y verás que ahora la base de datos ya tiene un par de tablas. Una es la tabla amigos creada por nuestra aplicación. 3.3 Vista

Queremos que al conectar un navegador con nuestro servidor aparezca una tabla con la lista de amigos, y que podamos usar el propio navegador para añadir amigos a esa lista, borrarlos o cambiar sus datos. Por tanto necesitamos que Flask genere vistas HTML para que puedan ser mostradas en el navegador.

El problema es que también querríamos que la misma aplicación pudiera servir como API REST para acceder a los datos y modificarlos desde otro tipo de aplicación (por ejemplo una aplicación móvil), en cuyo caso no queremos el HTML sino sólo los datos, preferiblemente en JSON. Para este cometido necesitamos que nuestra aplicación Flask implemente una API REST. ¿Podemos tener ambas cosas?

Siguiendo la filosofía REST, deberíamos usar los verbos HTTP GET, POST, PUT y DELETE sobre ciertas URLs que representen los recursos (los amigos), para significar acciones sobre la base de datos.

Sin embargo, un navegador web siempre hace peticiones GET (puede hacer POST si se trata de un formulario HTML). Nunca peticiones PUT o DELETE. Si queremos que el navegador consuma una API REST, el servidor debería enviarle código javascript que al ser ejecutado en el navegador invoque peticiones ajax. Esto complica innecesariamente la práctica, y hay una alternativa más sencilla.

Podemos hacer que Flask proporcione dos vistas:

  • Una vista HTML que un navegador pueda usar sin necesidad de JavaScript, y que se limite a verbos GET (y POST al enviar un formulario)
  • Otra vista que podemos llamar “API” que implemente una verdadera interfaz REST y que responda en JSON.

Flask hace sencillo implementar ambas, programadas de forma separada pero conviviendo en la misma app, bajo diferentes rutas, gracias al concepto de blueprints.

De momento nos centraremos en una única vista, la vista HTML, y más adelante introduciremos los blueprints e implementaremos la vista API.

3.3.1 Vista HTML

Para crear esta interfaz necesitamos hacer que Flask responda HTML cuando sea consultado sobre ciertas rutas. Además queremos que ciertas rutas causen modificaciones en la base de datos (altas, bajas, actualizaciones).

Por ejemplo, podríamos definir los siguientes verbos y rutas, para que lleven a cabo las acciones que se especifican a continuación:

  • GET /amigos Devuelve una tabla HTML en la que cada fila es un amigo, con su nombre, latitud y longitud. Cada fila de la tabla constará de un botón “modificar” y otro “borrar” que permitirán modificar o borrar los datos del amigo en cuestión. Al final de la tabla habrá otro botón “Añadir amigo” para poder introducir más elementos en la tabla.
  • GET /delete_amigo/<id> Elimina de la base de datos al amigo en cuestión, y devuelve la lista de amigos resultante.
  • GET /edit_amigo/<id> Devuelve un formulario HTML en el que ya vienen rellenados los datos del amigo, de modo que se pueda cambiar cualquiera de ellos. El formulario tiene un campo oculto con el id del amigo a modificar. El usuario no puede cambiar este valor porque no lo ve. El botón “Enviar” causará un POST a la URL /save_amigo, en el que se envían los datos del formulario, incluído el id oculto.
  • GET /new_amigo Presenta un formulario idéntico al anterior, sólo que todos los campos están vacíos (incluido el id oculto). El formulario tiene un botón “Enviar” que causa un POST a la URL /save_amigo, en el que se envían los datos que el usuario haya introducido en el formulario. El campo oculto id obviamente irá vacío.
  • POST /save_amigo Recibe los datos del formulario antes descrito. Si el campo oculto id va vacío, la acción a realizar será crear un nuevo elemento en la base de datos, con los valores leidos del formulario. Si el campo oculto id no es vacío la acción a realizar será una actualización del amigo con ese id en la base de datos. Tras realizar la modificación de la base de datos, ya sea el alta o la modificación, retorna la misma tabla que GET /amigos, en la que puede verse ya el nuevo amigo (si ha sido creado) o los datos modificados del amigo (si era una modificación).

Como se observa, la respuesta que ve el usuario ante cualquiera de esas acciones es siempre la tabla de amigos, por tanto debemos comenzar por implementar esa vista.

Una de las formas más comunes de implementar la vista es mediante plantillas (templates). Las plantillas permiten diseñar la vista HTML indicando donde se quiere que aparezca la información variable sin tener que ocuparse de cómo obtener dicha información. De este modo se puede separar el trabajo entre el diseño de la vista HTML (que puede encargarse a un especialista en diseño web para que le de un aspecto, colores y tipografía cuidados y acordes al estilo de la empresa), y la implementación de la funcionalidad en python.

3.3.2 La tabla de amigos

Hacer que Flask retorne HTML es sencillo. De la misma forma que retornamos "Hola mundo" en el ejemplo inicial, podemos retornar cualquier otra cadena, y por tanto HTML. Podríamos implementar una función python que genere el HTML a retornar, usando operaciones con cadenas, pues al final HTML es una cadena de texto. Pero hay formas mejores de hacerlo.

Flask soporta la idea de templates, que son ficheros HTML dentro de los cuales aparece un marcado especial para señalar qué variables Python deberían aparecer ahí. El marcado especial es un mini-lenguaje específico, y típicamente se usa jinja2 para esto.

<html>
  <head>
    <meta charset="utf-8">
    <title>Amigos</title>
  </head>
  <body>
  <h1>Amigos</h1>
    <table>
      <!-- Cabecera de la tabla -->
      <tr>
        <th>Nombre</th>
        <th>Latitud</th>
        <th>Longitud</th>
      </tr>
      <!-- Cuerpo de la tabla -->
      {% for amigo in amigos %} {# Esto es sintaxis jinja2 #}
      <tr>
        <td>{{amigo.name}}</td>
        <td>{{amigo.lati}}</td>
        <td>{{amigo.longi}}</td>
      </tr>
      {% endfor %}
    </table>
  </body>
</html>

4.1 Blueprints

La forma correcta de estructurar una aplicación Flask es mediante blueprints. Éstos se entienden como si fueran sub-aplicaciones, que aparecerán bajo diferentes rutas ante la aplicación general.

En nuestro caso, recordando que pretendíamos suministrar dos vistas, una en HTML para usar desde un navegador y otra API REST para consumir desde cualquier otro tipo de cliente, los blueprints nos permitirán separar esas funcionalidades en las rutas /html y /api, por ejemplo. No obstante el código forma parte de la misma aplicación, por lo que comparte el objeto db para acceder a la base de datos, el modelo de datos, y las funciones adicionales que queramos programar y que puedan ser útiles para ambas vistas. En esta aplicación tan sencilla no hay más funciones extra, pero una aplicación real toda la funcionalidad de negocio podría ser compartida entre ambas vistas.

Cada blueprint dará lugar a un subdirectorio dentro de app y cada uno de ellos contendrá su propio __init__.py y su propio views.py. No tienen su propio model.py porque el modelo de datos es común a ambas vistas.

La nueva estructura de carpetas será por tanto:

amigos/
    ├── app/
    │   ├── api/
    │   │   ├── __init__.py
    │   │   └── views.py
    │   ├── html/
    │   │   ├── __init__.py
    │   │   └── views.py
    │   ├── __init__.py
    │   ├── models.py
    │   └── templates/
    ├── config.py
    ├── instance/
    │   └── config.py
    ├── migrations/
    └── lanzar.py

La vista “api” no usa templates, esos sólo son necesarios para la vista “html”, por lo que decidimos no moverlos de donde estaban.

4.2 Blueprint html

El fichero app/html/__init__.py debe limitarse a declarar una variable de tipo Blueprint, con el nombre del blueprint en cuestión (html) e importar sus vistas. Para ello el código es:

from flask import Blueprint

html = Blueprint('html', __name__)

from . import views

El fichero app/html/views.py contiene las funciones que generarán las vistas de cada posible URL. De momento sólo habíamos hecho la vista para la url /amigos, que era la que generaba la tabla de amigos. Por tanto de momento tendremos sólo lo siguiente:

from flask import render_template
from . import html
from ..models import Amigo

@html.route("/amigos")
def tabla_amigos():
    """
    Obtiene la lista de amigos de la base de datos y la
    devuelve en una tabla HTML.
    """
    amigos = Amigo.query.all()
    return render_template("tabla_amigos.html",
                           amigos=amigos)

Observa lo siguiente:

  • Hemos llamado a la función tabla_amigos(), en vez de prueba(). El nombre no es muy importante pero ayuda a la legibilidad del código.
  • La ruta ante la que responde es /amigos y no /html/amigos. Esto es porque desde la app los blueprints se pueden “montar” en la ruta que se quiera. Después haremos que la app monte este blueprint en la ruta /html y así, aunque el cliente debe solicitar /html/amigos para obtener la tabla de amigos, en el blueprint html sólo tenemos que poner la parte /amigos, es decir, quitar el prefijo en el que está montada. Eso independiza el código del blueprint del punto de montaje.
  • Fíjate en los import iniciales. Se importa html (que es la variable instanciada en app/html/__init__.py) y para importar Amigo debe hacerse desde ..models

Finalmente, es necesario cambiar la función create_app(), puesto que ya no se definen dentro de ella rutas ni manejadores, sino que se limitará a importar el blueprint html y montarlo en la ruta /html.

El nuevo contenido de app/__init__.py será este:

from flask import Flask
from flask_sqlalchemy import SQLAlchemy
from flask_migrate import Migrate
from config import app_config

# Inicializar db, que conecta con la base de datos
db = SQLAlchemy()

def create_app(config_name):
    # La app se crea y configura como antes
    app = Flask(__name__, instance_relative_config=True)
    app.config.from_object(app_config[config_name])
    app.config.from_pyfile('config.py')
    db.init_app(app)
    migrate = Migrate(app, db)

    # Pero ahora no implementamos aquí funciones ni rutas
    # eso lo hacen los blueprints. Aquí simplemente registramos
    # esos blueprints para "montarlos" en una ruta
    from .html import html as html_blueprint
    app.register_blueprint(html_blueprint, url_prefix='/html')

    return app

4.2.2 Implementando acción “borrar”

Edita app/html/views.py para añadir una función que se ocupe de la ruta /delete_amigo/<id>. Observa que como parte de la ruta va el ID del amigo a eliminar. Flask tiene soporte para extraer ese id de la ruta y pasarlo como parámetro a la función, simplificando la implementación que queda tan simple como buscar en la base de datos ese amigo (devolver 404 si no existe) y borrarlo. Seguidamente queremos devolver la misma vista de la tabla de amigos, por lo que podemos hacer una redirección a la URL adecuada.

from flask import render_template, redirect, url_for
from .. import db
# Resto igual
# ...

@html.route("/delete_amigo/<int:id>")
def delete_amigo(id):
    """
    Borra un amigo de la base de datos
    """
    # El método get_or_404 proporcionado por SQLAlchemy
    # se ocupa de generar un error 404 si el id no está en
    # la base de datos
    amigo = Amigo.query.get_or_404(id)
    # Una vez obtenido, lo borramos
    db.session.delete(amigo)
    db.session.commit()

    # Y redireccionamos a la vista /amigos
    return redirect(url_for('html.tabla_amigos'))

Observa cómo finaliza la función. El método redirect() de Flask envía un código 302 al cliente con la URL que le pases como parámetro. El navegador, al recibir el 302 hará otro GET a la URL indicada y así cargará otra página. Pero observa que hemos evitado pasarle una URL concreta (como sería /html/amigos), sino que hemos optado por usar la función url_for() a la que se le pasa el nombre de una función dentro de un módulo (en este caso módulo html, función tabla_amigos) para que sea Flask quien averigue qué ruta corresponde a esa función. Esto es más flexible pues nos permite cambiar el punto de montaje del Blueprint, o el nombre de la ruta dentro del mismo, sin romper nada.

4.2.3 Implementando las acciones de editar/añadir amigo

Necesitamos un nuevo template, que será el formulario HTML con los datos del amigo en cuestión (o en blanco si es un amigo nuevo). No es objetivo de la asignatura explicar cómo se escribe un formulario HTML. Simplemente comentar que en este caso, el formulario tendrá tres campos visibles para el usuario: nombre, latitud y longitud del amigo, y uno invisible, el id del amigo. El id es necesario para que cuando el servidor reciba los datos sepa qué amigo debe modificar en la base de datos, pero no debe ser visible al usuario porque éste no puede cambiar el id de un elemento de la base de datos.

Habría muchas más cosas que explicar sobre formularios, tales como la inclusión de comprobaciones en el lado cliente (por ejemplo, limitar el número de caracteres de un campo, forzar a que el usuario deba poner algo en los campos obligatorios, protegerse de ataques CSRF, etc. Flask proporciona el paquete flask-wtf para simplificar la creación de formularios seguros, pero no la usaremos aquí).

Escribe lo siguiente en app/templates/edit_amigo.html:

<html>
  <head>
    <meta charset="utf-8">
    <title>Amigo - Modificar datos</title>
  </head>
  <body>
  <h1>Amigo - Modificar datos</h1>
  <form action="/html/save_amigo" method="POST">
    <input type="hidden" name="id" value="{{amigo.id}}"/>
    <label for="name">Nombre</label>
    <input type="text" name="name" value="{{amigo.name}}"></br>
    <label for="longi">Logitud</label>
    <input type="text" name="longi" value="{{amigo.longi}}"></br>
    <label for="lati">Latitud</label>
    <input type="text" name="lati" value="{{amigo.lati}}"></br>
    <input type="submit" value="Enviar"></form>
  </body>
</html>

Lo más importante del código anterior es la línea <form action=...>, pues es la que especifica qué ocurrirá cuando el usuario pulse el botón “Enviar”, una vez haya modificado a su antojo los campos del formulario. Y en esa línea estamos diciendo que haga una petición POST (sólo se permite POST o GET), a la url /html/save_amigo, donde habrá otra función python (aún por escribir) que guarde los cambios en la base de datos.

Este template que acabamos de escribir nos sirve tanto para el caso en que estemos editando los datos de un amigo concreto, como para el caso en que estemos creando datos para un amigo nuevo. El formulario es el mismo, sólo que la variable amigo no tendría ningún valor en el segundo caso (valdrá None).

Ese template nos sirve para responder tanto a la petición GET /html/edit_amigo/<id> como a GET /html/new_amigo. Edita app/html/views.py y añade las funciones siguientes:

@html.route("/edit_amigo/<int:id>")
def edit_amigo(id):
    """
    Presenta un formulario para obtener datos a modificar de un amigo
    """
    amigo = Amigo.query.get_or_404(id)
    # Rellenar el template con los datos de este amigo y presentar
    # el formulario al usuario
    return render_template("edit_amigo.html", amigo=amigo)

@html.route("/new_amigo/")
def new_amigo():
    """
    Presenta un formulario para obtener datos para crear nuevo amigo
    """
    # No tenemos datos de ningún amigo previo
    # Pero podemos usar el mismo template
    return render_template("edit_amigo.html", amigo=None)

Fíjate que la función new_amigo() hace uso del mismo template que la función edit_amigo(), pero haciendo amigo=None.

Quizás podrías pensar que eso causará errores al renderizar el template, puesto que si amigo es None ¿qué pasará al intentar acceder a amigo.id o amigo.name? El temor es razonable, pero jinja2 es lo bastante listo como para dejar en blanco esos valores en caso de que su evaluación produzca excepciones como es el caso. Por tanto el usuario verá un formulario completamente vacío donde poder introducir información sobre el nuevo amigo.

Y lo que es mejor, el campo oculto id también tiene como valor la cadena vacía. Esto permite usar la misma URL en el POST, tanto para modificar un amigo, como para crear uno nuevo. La función que reciba los datos del formulario sabrá si se trata de un nuevo amigo o de uno existente gracias al campo oculto id del formulario.

Para completar esta funcionalidad sólo queda escribir la función que responda al POST /html/save_amigo. Esta función deberá extraer los datos del formulario (que recibirá en el cuerpo de la petición) y decida en base a ellos si actualizar un amigo de la base de datos o crear uno nuevo.

Para acceder a los datos del formulario debemos importar de flask la variable request, que es una especie de variable global (pero única para cada petición recibida) que nos dará acceso a los datos del FORM. Debemos validar esos datos (por ejemplo, si el usuario no ha introducido nombre, o si el nombre ya existía, rechazaremos la creación del nuevo amigo). Una forma sencilla de generar errores es usar la función abort de flask, que importaremos también.

Edita app/html/views.py y añade la siguiente función:

# Añadir nuevos imports en la primera línea
from flask import render_template, redirect, url_for, request, abort

# ...
@html.route("/save_amigo", methods=["POST"])
def save_amigo():
    id = request.form.get("id")
    if id is None or id == "":
        # En este caso se trata de un amigo nuevo, lo creamos
        name = request.form.get("name")
        if not name:
            # Si falta el nombre, devolvemos un error HTTP 422
            abort(422)
        # Para lati y longi permitimos que estén ausentes, y en ese caso
        # les damos un valor por defecto de "0"
        lati = request.form.get("lati", "0")
        longi = request.form.get("longi", "0")

        # Creamos el amigo y lo añadimos a la base de datos
        amigo = Amigo(name=name, lati=lati, longi=longi)
        db.session.add(amigo)
        db.session.commit()
    else:
        # En este caso se trata de un amigo de la base de datos
        amigo = Amigo.query.get_or_404(int(id))
        # Modificamos los campos que vengan en el formulario
        name = request.form.get("name")
        if name:
            amigo.name = name
        lati = request.form.get("lati")
        if lati:
            amigo.lati = lati
        longi = request.form.get("longi", "0")
        if longi:
            amigo.longi = longi
        # Una vez modificado, lo guardamos a la base de datos
        db.session.commit()
    # Redireccionamos hacia la tabla-lista de amigos
    return redirect(url_for("html.tabla_amigos"))

Observa los siguientes aspectos de este código:

  • Para leer los datos que llegan del formulario, usamos la variable global request, que es la petición enviada por el navegador, y dentro de ella su campo form que es un diccionario python cuyas claves son las que hayamos puesto en el formulario y los valores los introducidos por el usuario.
  • Podríamos acceder por ejemplo al nombre del amigo con request.form["name"], pero esto podría fallar si el cliente no ha enviado ese campo (aunque tal cosa no puede pasar si el cliente es el navegador web usando nuestra propia página, pero podría ocurrir desde un cliente mal programado o malintencionado). Para evitar posibles problemas usamos request.form.get("name"), que no rompe si "name" no está en el formulario, sino que devuelve None.
  • Observa cómo comprobamos que "id" esté en el formulario y si no está o es vacío, creamos un nuevo amigo.
  • El campo "name" lo hacemos obligatorio. Si no viene, generamos un error 422 (HTTP Unprocessable Entity). Los campos "longi" y "lati" los hacemos opcionales y si no vienen les damos el valor por defecto "0". Usando .get() es tan sencillo como pasar un segundo parámetro que sería el valor por defecto.
  • Los comandos para crear el amigo y añadirlo a la base de datos son iguales a los que hicimos cuando creamos “a mano” los dos primeros amigos de ejemplo, desde un intérprete python.
  • En caso de que venga un "id", lo usamos para sacar de la base de datos a ese amigo (o generamos un error 404 si no existe), y modificamos en él los campos que nos lleguen del formulario.

5 Tercera versión de la aplicación

De la sección anterior tenemos listo un servidor que es apropiado para interactuar con un usuario humano, a través de un navegador Web, pero no es apropiado para interactuar con un cliente que sea otra aplicación (por ejemplo, si queremos ver o modificar la lista de amigos desde línea de comandos, a través de un script en python, o desde una aplicación móvil desarrollada en Java o Kotlin).

Para lo segundo, tenemos que programar una versión del servidor que en lugar de responder con HTML, responda JSON, que es más fácil de parsear, y que en lugar de recibir los datos en un form (formulario HTML), los reciba también en JSON. Y preferiblemente, para acercarnos más a la filosofía REST, que use correctamente los verbos HTTP, es decir, que use DELETE en lugar de GET cuando quiera eliminar un amigo de la base de datos, o PUT en lugar de POST cuando quiera modificar uno. También debe usar nombres de recursos que no incluyan el verbo. Por ejemplo, borrar un amigo cuyo id es 1 debería hacerse con la petición HTTP DELETE /amigo/1, en lugar de usar GET /delete_amigo/1 como hacíamos en la versión HTML (donde no quedaba otro remedio que hacerlo así porque el navegador Web no puede emitir peticiones DELETE).

También nos será útil (por razones relacionadas con otra práctica futura), tener una forma de encontrar a un amigo en la base de datos por su nombre, en lugar de por su id. Por ejemplo, poder hacer un GET /amigo/byName/Manolo.

Detallaremos por tanto a continuación qué URLs compondrán la API y qué verbos HTTP se pueden usar en cada una de ellas.

5.2 Implementación de la API REST

Flask es perfecto para implementar este tipo de APIs. Ya lo hemos usado antes para la interfaz html, y entonces hicimos uso de templates y de jinja2 para retornar HTML. 5.2.6 Implementación de PUT /amigo/id

Cuando se recibe el verbo PUT sobre la ruta que identifica a un amigo concreto (por su id), hay que actualizar los datos de ese amigo, que son recibidos a través de request.json. El siguiente código te muestra cómo:

Edita app/api/views.py y añade esta función:

@api.route("/amigo/<int:id>", methods=["PUT"])
def edit_amigo(id):
    """
    Modifica en la base de datos el amigo cuyo id recibe como parámetro.

    Retorna el JSON con el amigo tras la modificación
    """
    # Obtenemos el amigo a partir de su ID
    amigo = Amigo.query.get_or_404(id)

    # Comprobamos que hemos recibido JSON como parte del PUT
    if not request.json:
        abort(422, "No se ha enviado JSON")

    # Intentamos extraer campos del JSON (si no están presentes)
    # la extracción retornará None
    name = request.json.get("name")
    lati = request.json.get("lati")
    longi = request.json.get("longi")
    # Usamos los campos que estén presentes para actualizar el objeto amigo
    if name:
        amigo.name = name
    if lati:
        amigo.lati = lati
    if longi:
        amigo.longi = longi

    # Finalmente, si hemos cambiado algo en el objeto amigo, hacemos
    # el commit a la base de datos para que se guarden las modificaciones
    if name or lati or longi:
        db.session.commit()

    # Y retornamos el JSON con los nuevos datos
    amigodict = {"id": amigo.id, "name": amigo.name,
                 "longi": amigo.longi, "lati": amigo.lati }
    return jsonify(amigodict)

El comando http permite que especifiquemos nombres de campos y valores en la línea de comandos mediante campo=valor, y con la opción --json se ocupa de generar un JSON con esa información que es enviada al servidor en el cuerpo del PUT.

Fíjate que la forma de acceder a los parámetros es la misma que la que ya usamos para el PUT, pero en esta ocasión debemos asegurarnos de que se proporciona un nombre y de que éste no existe ya en la base de datos. Fíjate cómo lo hacemos:

@api.route("/amigos", methods=["POST"])
def new_amigo():
    """
    Modifica en la base de datos añadiendo un amigo cuyos datos
    recibe en JSON.

    Retorna el JSON con el amigo tras la creación, para que el cliente
    pueda conocer el id del amigo recién creado.
    """

    # Comprobamos que hemos recibido JSON como parte del PUT
    if not request.json:
        abort(422, "No se ha enviado JSON")

    # Intentamos extraer campos del JSON (si no están presentes)
    # la extracción retornará None
    name = request.json.get("name")
    # Si no hay nombre, rechazamos la petición
    if not name:
        abort(422, "El JSON no incluye el campo 'name'")

    # Si el nombre ya existe en la base de datos, también lo rechazamos
    amigo = Amigo.query.filter_by(name = name).first()
    if amigo:
        abort(422, "Ya existe un amigo con ese nombre")

    # En caso contrario, tomamos latitud y longitud. Si no vienen les
    # damos un valor de 0
    lati = request.json.get("lati", "0")
    longi = request.json.get("longi", "0")

    # Creamos un nuevo amigo con esos datos
    amigo = Amigo(name=name, lati=lati, longi=longi)
    db.session.add(amigo)
    db.session.commit()

    # Y retornamos el JSON con los datos del nuevo amigo
    amigodict = {"id": amigo.id, "name": amigo.name,
                 "longi": amigo.longi, "lati": amigo.lati }
    return jsonify(amigodict)

Este consiste en activar la imagen que hemos creado en el apartado anterior, pero pasándole como variables de entorno las que necesita para conocer el modo de despliegue (usaremos “production” ahora) y la URL a la base de datos (usaremos ahora el nombre del nodo “basedatos” en lugar de su IP, gracias al servidor DNS incluido en docker).

El comando es por tanto:

$ docker run --name amigos \
   -e DEPLOYMENT_MODE=production \
   -e DATABASE_URI=mysql+pymysql://amigosuser:amigospass@basedatos/amigosdb \
   --rm -d --network pruebas amigos:1.0

Comprueba con docker ps que está en ejecución. No podemos conectar a él porque el puerto 5000 en que escucha es interno al contenedor. Hasta que no despleguemos nginx que actuará de proxy, será inaccesible.

6.3.3 Contenedor con nginx

El despliegue de nginx es más simple que el usado en la sesión anterior porque en esta ocasión no tiene que acceder a carpetas html con contenidos estáticos, y por tanto no necesitamos montar esos volúmenes, sino tan sólo el que contiene default.conf. Importante la redirección del puerto 80 para que sea accesible desde el exterior.

$ docker run --rm -d --network pruebas --name nginx \
       -v $(pwd)/sitios_nginx:/etc/nginx/conf.d \
       -p 80:80 nginx

6.4 Comprobación final

Conecta el navegador a la IP de tu máquina ubuntu (no necesitas especificar puerto, irá por defecto al 80) y ve a la ruta /html/amigos. Si todo ha ido bien, verás la tabla de amigos. Desde aquí puedes añadir, borrar o modificar amigos.

Usando el mismo navegador ve a /api/amigos y verás ahora la lista en JSON. Desde aquí no puedes añadir, borrar o modificar los datos, puesto que el navegador no puede hacer peticiones PUT ni DELETE, y los POST los envía sólo ante la acción submit de un formulario. Esta api será usada más adelante en una práctica en la que el cliente será una aplicación Android.

7 Ejercicios adicionales

7.1 Dispositivos de los amigos

Como parte de la futura práctica Android, cada “amigo” de la base de datos necesita tener un campo nuevo, llamado device, que será una cadena de 4096 caracteres. Estas cadenas tan largas se almacenan en un campo de tipo TEXT (dentro de SQLAlchemy sería db.Text()). Por defecto estas cadenas estarán vacías (es decir, contienen NULL en la base de datos, None al leerlas en python), pero la API REST debe permitir cambiarlas (mediante un PUT al correspondiente amigo).

Ejercicio

Usando el servidor de desarrollo flask para probar las modificaciones que vayas haciendo:

  1. Realiza las modificaciones apropiadas en el modelo para incorporar el nuevo campo device.
  2. Realiza la migración y el upgrade de la base de datos.
  3. Modifica las vistas, tanto html como api para que muestren el campo device de cada amigo en las operaciones GET.
  4. Modifica las vistas adecuadas para que se pueda modificar el valor del dispositivo, tanto a través del formulario HTML (ten en cuenta la observación anterior sobre el valor "None") como a través de un PUT al amigo correspondiente. La aplicación debe aceptar que venga una cadena vacía en el campo device, y la guardará en la base de datos. Para las vistas jinja del formulario, si te limitas a usar el valor de {{amigo.device}}, saldrá la cadena "None" en el navegador, y si el usuario no la cambia mandará de vuelta esa cadena al servidor, que guardaría entonces la cadena "None" en la base de datos, en lugar de una cadena vacía. Queremos por tanto que si en la base de datos el device era nulo, en el formulario salga la cadena vacía, por lo que deberás poner {{amigo.device or ''}} para lograrlo.
  5. Añade a la API una funcion que responda ante un GET /api/devices suministrando la lista de todos los device no nulos (ni cadenas vacías) en la base de datos.

Una vez has probado con el servidor de desarrollo que todo funciona correctamente:

  • Finalmente rehaz (docker build) la imagen amigos con el tag amigos:2.0 y prueba a lanzar todo en contenedores.

  • Asegúrate de guardar en git no sólo los cambios que hayas hecho al código, sino también las migraciones creadas por flask-migrate, el Dockerfile (si has tenido que cambiarlo, en principio no hace falta) y los scripts que hayas modificado para lanzar la nueva versión.

Tras haber hecho tus experimentos, es posible que hayas dejado “basura” en los campos device de los amigos de la base de datos. Edítalos para que todos ellos sean la cadena vacía, pues de lo contrario habrá problemas en la práctica de Android.
